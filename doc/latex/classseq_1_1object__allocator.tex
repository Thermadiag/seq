\doxysubsection{Detailed Description}
\subsubsection*{template$<$class Pool$>$\newline
class seq\+::object\+\_\+allocator$<$ Pool $>$}

Stl conforming allocator based on an object pool class. 


\begin{DoxyTemplParams}{Template Parameters}
{\em Pool} & object pool type, either \#object\+\_\+pool or \#parallel\+\_\+object\+\_\+pool\\
\hline
\end{DoxyTemplParams}
\mbox{\hyperlink{classseq_1_1object__allocator}{seq\+::object\+\_\+allocator}} is a stl compliant allocator dedicated to node based containers like std\+::set, std\+::map, std\+::list, std\+\_\+deque, std\+::multimap, std\+::multiset... \mbox{\hyperlink{classseq_1_1object__allocator}{object\+\_\+allocator}} also works for other containers like std\+::vector, but does not provide any benefits in such cases.

It provides faster allocation/deallocation time as well as reduced memory footprint and reduced memory fragmentation (see \mbox{\hyperlink{classseq_1_1object__pool}{seq\+::object\+\_\+pool}} class for more details). Since the \mbox{\hyperlink{classseq_1_1object__pool}{seq\+::object\+\_\+pool}} class uses chunk based allocation, containers using \mbox{\hyperlink{classseq_1_1object__allocator}{object\+\_\+allocator}} are usually more cache friendly, which fasten most operations(like iterating, sorting, ...).

Note that \mbox{\hyperlink{classseq_1_1object__allocator}{object\+\_\+allocator}} does not work with std\+::list\+::sort() on gcc. (\href{https://stackoverflow.com/questions/63716394/list-sort-fails-with-abort-when-list-is-created-with-stateful-allocator-when-com}{\texttt{ https\+://stackoverflow.\+com/questions/63716394/list-\/sort-\/fails-\/with-\/abort-\/when-\/list-\/is-\/created-\/with-\/stateful-\/allocator-\/when-\/com}})

Usage example for std\+::list\+:

\DoxyCodeLine{\textcolor{keyword}{using} allocator = object\_allocator< object\_pool<int> >;}
\DoxyCodeLine{}
\DoxyCodeLine{std::list<int, allocator > my\_list;}
\DoxyCodeLine{\textcolor{comment}{//....}}
\end{DoxyCode}
